# v0.5 Safe Fix Mode Design

**Date:** 2026-06-19
**Phase:** v0.5 (per `docs/PRODUCT_PLAN.md`)
**Status:** Approved

## Goal

Fill the existing `fix` command so it scans readiness notes and applies safe patches to 4 classes of source-repo issues. Default is dry-run (prints diffs); `--apply` writes changes. Fixes are labeled by certainty so users know what is safe.

## Scope (in)

- `repo-brief fix` subcommand: scans repo, runs 4 fixers, prints diffs (dry-run) or writes patches (`--apply`)
- 4 fixers: `fixPackageManagerField`, `fixReadmePackageManagerCommands`, `fixCiPackageManagerCommands`, `fixMissingStandardScripts`
- Zero-dep line-level diff renderer (LCS algorithm)
- Certainty labels: `deterministic` (3 fixers) vs `assumed-standard` (lint/typecheck scripts)
- Chain-patch mechanism: multiple fixers editing the same file are merged into one final patch
- Unit + integration tests
- Docs: README fix section, PRODUCT_PLAN check-off with role clarification
- `0.5.0` release

## Scope (out)

- Generating brief files (still `brief` command's job — PRODUCT_PLAN's "generate missing context files" and "update generated files" sub-items are satisfied by `brief`)
- Deleting lockfiles (Multiple lockfiles note is report-only; which to delete is ambiguous)
- Adding test/build scripts (content is ambiguous, risky)
- Fixing "README references nonexistent script" / "CI references nonexistent script" (fix README vs add script is ambiguous)
- Auto `git add` / `git commit` (user controls git workflow)
- `--dry-run` flag on fix (fix is dry-run by default; flag would be redundant)
- File backups before `--apply` (rely on user's git working tree for rollback)

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| New `src/fix/` module, scanner untouched | Clean separation of detect (scanner) vs fix (fixer) vs apply. Minimal blast radius — scanner/render/MCP/types unchanged |
| Fixers are pure functions: `(brief, fileContents) => Fix[]` | Easy TDD; no I/O in the fixer itself, all file reads happen up front |
| Regex-based file patching, NOT JSON.parse → stringify | JSON re-serialization would destroy formatting/indentation/key order. Regex preserves the rest of the file byte-for-byte |
| Fix carries `originalContent` + `patchedContent` | Diff rendering and apply are decoupled from fix logic |
| Certainty label per fix | 3 deterministic fixers vs 1 assumed-standard (lint/typecheck uses conventional commands). Users see the difference in dry-run output |
| Chain-patch: same-file fixes merged | If fixer A and B both edit package.json, B's input is A's output. Final Fix has cumulative patch |
| Default dry-run, `--apply` to write | Honors "safe" in the phase name; zero default destruction |
| Direct write on apply, rely on git for rollback | Standard CLI convention; matches how developers use any patching tool |
| Dynamic `import()` for fix module in cli.ts | Keeps fix module out of brief/doctor/mcp hot path; consistent with v0.4's mcp wiring |

## Architecture

### Data Flow

```
repo-brief fix [--apply]
   ↓
scanRepository(cwd) → brief
   ↓
runFixers(brief, fileContents) → Fix[]
   Each Fix = { filePath, certainty, note, originalContent, patchedContent }
   ↓
Chain-merge: group by filePath, fold patches so each file has one final Fix
   ↓
apply=false → renderDiffs(fixes) → print to stdout (default)
apply=true  → applyFixes(fixes) → write files + print summary + rollback hint
```

### Components

1. **`src/fix/types.ts`** — `Fix`, `Certainty`, `Fixer`, `FixerContext`, `ApplySummary`
2. **`src/fix/diff.ts`** — LCS line-diff renderer; `renderDiff(original, patched): string`
3. **`src/fix/fixers.ts`** — 4 fixer functions + `runFixers` (collect + chain-merge)
4. **`src/fix/apply.ts`** — `renderDiffs(fixes): string` + `applyFixes(fixes): Promise<ApplySummary>`
5. **`src/fix/index.ts`** — `runFixCommand(cwd, options, io): Promise<number>` orchestration
6. **`src/cli.ts`** — `parseArgs` gains `--apply`; `fix` branch calls `runFixCommand` via dynamic import

## Fixer Specifications

All fixers take `(brief, fileContents)` where `fileContents: Map<string, string>` is pre-read file content. Fixers return `Fix[]`.

### Fixer 1: `fixPackageManagerField` (deterministic)

- **Trigger:** `brief.readinessNotes` contains `packageManager field declares X, but lockfile suggests Y.`
- **Action:** Read `package.json`, change `packageManager` field value from `X@<version>` to `Y@<version>`, preserving version.
- **Regex:** `/("packageManager"\s*:\s*")[^"]+(")/` → `$1${newManager}@${version}$2`
- **Does not trigger:** when `packageManager` field is absent (scanner never reports this note in that case)

### Fixer 2: `fixReadmePackageManagerCommands` (deterministic)

- **Trigger:** `brief.readinessNotes` contains `README mentions X commands, but Y suggests Z.`
- **Action:** Replace wrong package-manager command prefixes in README with the correct one.
- **Replacement rules** (target = lockfile manager):
  - Excluded from replacement (scanner's `PACKAGE_MANAGER_COMMANDS`): `install`, `add`, `remove`, `exec`, `dlx`, `ci`
  - Target npm: `pnpm run X`→`npm run X`, `pnpm X`→`npm run X`, `yarn X`→`npm run X` (for X∈{test,build,lint,dev}: `yarn test`→`npm test`), `bun run X`→`npm run X`
  - Target pnpm: `npm run X`→`pnpm X`, `npm test`→`pnpm test`, `npm build`→`pnpm build` (pnpm needs no `run`)
  - Target yarn: `npm run X`→`yarn X`, `npm test`→`yarn test`
  - Target bun: `npm run X`→`bun run X`, `npm test`→`bun test`
- Global regex per rule; every occurrence replaced.

### Fixer 3: `fixCiPackageManagerCommands` (deterministic)

- **Trigger:** `brief.readinessNotes` contains `GitHub Actions uses X, but lockfile suggests Y.`
- **Action:** Iterate `.github/workflows/*.yml`; on `- run: <cmd>` / `run: <cmd>` lines, replace wrong package-manager prefix with correct one.
- **Replacement rules:** same mapping as Fixer 2, scoped to the `runValue` portion of each run line.

### Fixer 4: `fixMissingStandardScripts` (assumed-standard)

- **Trigger:** `brief.readinessNotes` contains `ESLint config found but no lint script` OR `tsconfig.json found but no typecheck script`
- **Action:** Insert standard script into `package.json` `scripts`.
  - eslint config + no `lint` → add `"lint": "eslint ."`
  - tsconfig + no `typecheck` → add `"typecheck": "tsc --noEmit"`
- **Insertion:** regex-locate `"scripts": { ... }` block; insert before closing `}`.
- **Certainty:** `assumed-standard` (assumes conventional commands; user may have custom config wanting different commands). Labeled in diff output.

## Diff Rendering (`diff.ts`)

Zero-dep line-level diff via LCS (Longest Common Subsequence).

- Input: `originalContent`, `patchedContent` (strings)
- Algorithm: split by line, LCS DP, backtrack to produce unified diff
- Output format: `--- a/<path>` / `+++ b/<path>` header, `@@ -oldStart,oldLen +newStart,newLen @@` hunks, `  ` context / `- ` removed / `+ ` added
- Complexity: O(n*m) acceptable for typical config files (< 500 lines)

## Apply Logic (`apply.ts`)

### `renderDiffs(fixes: Fix[]): string`

Renders each fix as a labeled block:
```
[certainty] <filePath>
  reason: <corresponding scanner note>
  <unified diff>
```
The label prefix (`[deterministic]` / `[assumed-standard]`) lets users distinguish safe patches from assumed-standard ones at a glance.

### `applyFixes(fixes: Fix[]): Promise<ApplySummary>`

- For each Fix, `writeFile(filePath, patchedContent)`
- Returns `{ applied: number, files: string[] }`
- Chain-merge happens in `runFixers` BEFORE applyFixes sees fixes (so applyFixes gets one final Fix per file)

### Chain-patch mechanism (in `runFixers`)

After collecting raw Fix[] from all fixers:
1. Group by `filePath`
2. For each group with multiple fixes, fold: `finalPatched = fixes.reduce((acc, f) => applyPatchToContent(f.originalContent, f.patchedContent, acc), originalRawContent)`
3. Simplification: since each fixer already produced a `patchedContent` from the SAME `originalContent`, chain-merging is actually: take the first fix's `patchedContent` as input to the second fixer's transformation. **Cleaner approach:** fixers run sequentially, each one operating on the CURRENT state of fileContents (updated in place). So fixer 2 sees the result of fixer 1.

**Chosen chain approach:** `runFixers` maintains a mutable `fileContents` Map. Each fixer reads from and writes back to this Map. After all fixers run, for each file whose content changed from original, emit one Fix with `originalContent = original snapshot` and `patchedContent = current Map value`. This guarantees correct chaining and one Fix per file.

## CLI Integration (`cli.ts`)

### `parseArgs` extension

```typescript
interface ParsedArgs {
  command: "brief" | "doctor" | "fix" | "mcp";
  cwd: string;
  dryRun: boolean;
  apply: boolean;  // fix-only
}
```

Parse `--apply` → `apply = true`. `fix` does NOT accept `--dry-run` (it's dry-run by default).

### `runCli` fix branch

Replaces the current placeholder:
```typescript
if (parsed.command === "fix") {
  const { runFixCommand } = await import("./fix/index.js");
  return runFixCommand(parsed.cwd, { apply: parsed.apply }, io);
}
```

## Error Handling & Edge Cases

| Situation | Behavior |
|-----------|----------|
| No readiness notes trigger any fixer | Print `No fixes needed. Your repo looks clean.` Exit 0 |
| Dry-run with fixes | Print diffs. Exit 0 |
| `--apply` with fixes | Write + print summary + rollback hint. Exit 0 |
| File read failure (permissions) | Fixer skips that file, prints warning, doesn't block others |
| Regex doesn't match (file already manually fixed) | Fixer produces 0 Fixes for that case, silent skip |
| package.json not valid JSON but fixer needs to patch | Fixer uses regex (no JSON.parse), so handles malformed JSON; if regex doesn't match, skips |
| Cross-line package-manager commands in README/CI | Not handled (scanner also reads single-line), stays consistent |
| Multiple fixers edit same file | Chain via mutable fileContents Map (one final Fix per file) |

## Out-of-Scope Behaviors (explicit)

- Does NOT generate brief files (brief command's job)
- Does NOT delete lockfiles (ambiguous which to delete)
- Does NOT add test/build scripts (ambiguous content)
- Does NOT fix "README references nonexistent script" / "CI references nonexistent script" (fix README vs add script is ambiguous)
- Does NOT auto `git add`/`git commit`
- Does NOT accept `--dry-run` (fix is dry-run by default)
- Does NOT backup files before `--apply` (rely on git)

## Testing Strategy

TDD: write failing test, implement, verify, per fixer.

### `tests/fix/diff.test.ts`
- Identical contents → empty diff
- Single line change → correct hunk header + `-`/`+` lines
- Multi-line add/remove → correct LCS result
- 100-line file with 1 change → hunk covers only changed region

### `tests/fix/fixers.test.ts`

**fixPackageManagerField:**
- note present + field matches regex → produces Fix, version preserved
- note absent → 0 Fixes
- field format abnormal (no @version) → regex no match → 0 Fixes

**fixReadmePackageManagerCommands:**
- npm→pnpm: `npm run build`→`pnpm build`, `npm test`→`pnpm test`
- install-class commands NOT replaced: `npm install` unchanged
- note absent → 0 Fixes

**fixCiPackageManagerCommands:**
- `- run: npm run test` → `- run: pnpm test`
- `uses:` lines untouched
- note absent → 0 Fixes

**fixMissingStandardScripts:**
- eslint config + no lint → adds `"lint": "eslint ."`
- tsconfig + no typecheck → adds `"typecheck": "tsc --noEmit"`
- script already present → 0 Fixes

### `tests/fix/apply.test.ts`
- `renderDiffs` single Fix → output contains `[deterministic]` label + diff
- `renderDiffs` assumed-standard Fix → output contains `[assumed-standard]` label
- `applyFixes` → writes temp file, verifies content + returns correct summary
- Chain-patch test: two fixers edit same file → merged into single final Fix with both changes

### `tests/fix/index.test.ts` (end-to-end)
- Temp dir fixture with multiple issues (npm-mentioned README + pnpm-lock + wrong packageManager field + eslint config without lint script)
- `runFixCommand` dry-run → multiple fixes detected
- `runFixCommand({ apply: true })` → files actually changed + content verified

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Regex replacement injures non-command README text | Medium | Wrong text changed | Strict matching of scanner command boundaries; tests cover install-class exclusion |
| Chain-patch order invalidates later patches | Low | Later fixer's offset stale | Fixers work on current content (not offsets); mutable fileContents Map guarantees correct chaining |
| JSON re-serialization destroys formatting | High | package.json indent/order lost | ALL patches use regex, never JSON.parse→stringify |
| assumed-standard script conflicts with user config | Medium | Adds unwanted lint command | `assumed-standard` label + default dry-run double protection |
| Fix runs on file without git commit | Low | Second run finds no fix | Normal; regex won't match already-fixed content, silent skip |
