# v0.4 MCP Server Design

**Date:** 2026-06-17
**Phase:** v0.4 (per `docs/PRODUCT_PLAN.md`)
**Status:** Approved

## Goal

Add a stdio MCP server as a new `repo-brief mcp` subcommand. Coding agents query structured repo context (repo map, commands, readiness notes) on demand instead of reading every generated markdown file.

## Scope (in)

- `repo-brief mcp` subcommand launching a stdio JSON-RPC 2.0 MCP server
- 4 MCP tools: `get_repo_map`, `get_commands`, `get_readiness_notes`, `refresh`
- Scan-on-startup + in-memory cache; `refresh` tool rescans
- Zero runtime dependencies (Node stdlib only: `readline`, `process.stdout`)
- Unit tests for protocol layer (mock stdin/stdout)
- Docs: README MCP config examples (Claude Code, Cursor); PRODUCT_PLAN check-off
- `0.4.0` release

## Scope (out)

- HTTP/SSE transport (YAGNI for single-repo, single-agent)
- Official `@modelcontextprotocol/sdk` dependency (keep zero-deps style)
- Separate npm package (subcommand in same package)
- Refresh accepting a path argument (cwd is fixed at startup)
- Changes to `brief` / `doctor` / `fix` commands (v0.4 is MCP-only)
- Scanner optimization (out of scope per scope guardrails)

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| stdio transport | Default for Claude Code / Cursor / Codex; no ports, no auth, no network |
| Scan-on-startup + cache | Fast tool calls; `refresh` tool solves staleness on agent's terms |
| 4 domain tools aligned to PRODUCT_PLAN wording ("repo map, commands, readiness notes") | Tool boundaries map 1:1 to docs; easy to explain; agent picks the domain it needs |
| Return structured JSON, not markdown | Core MCP value vs. reading files — agent parses directly, saves tokens |
| Zero-dep hand-written JSON-RPC subset | Matches repo-brief's zero-runtime-deps style; only `initialize` / `tools/list` / `tools/call` needed |
| Subcommand `mcp` in same binary | Zero extra publishing burden; agent config is `npx @rogerdigital/repo-brief mcp` |
| `process.cwd()` as scan root (fixed at startup) | Simplest, matches stdio MCP convention |

## Architecture

### Data Flow

```
agent host launches: npx @rogerdigital/repo-brief mcp
   ↓
server startup: scanRepository(process.cwd()) → cache (RepositoryBrief)
   ↓
host sends initialize → server replies capabilities (stdio, 4 tools)
   ↓
host sends tools/list → server replies 4 tool schemas
   ↓
host sends tools/call {name: "get_repo_map"} → server reads cache → returns structured JSON
   ...
host sends tools/call {name: "refresh"} → server rescans → updates cache → returns {scannedAt, root}
   ↓
host closes stdin → server process exits
```

### Components

1. **`src/mcp/server.ts`** — MCP protocol layer (~150 lines)
   - JSON-RPC frame reader (newline-delimited JSON over stdin via `readline`)
   - Response writer (`process.stdout.write` with trailing newline)
   - Dispatch on `method`: `initialize` / `tools/list` / `tools/call`
   - Exports `runMcpServer(cwd: string): Promise<void>`
2. **`src/mcp/tools.ts`** — Tool definitions and executors
   - `TOOL_DEFINITIONS`: array of `{name, description, inputSchema}` for `tools/list`
   - `executeTool(name, cache): ToolResult` — reads from cached `RepositoryBrief`
   - `refreshCache(cwd): Promise<RepositoryBrief>` — wraps `scanRepository`
3. **`src/cli.ts`** — add `mcp` to `ParsedArgs["command"]` union; in `runCli`, branch to `runMcpServer(parsed.cwd)` and return early (MCP owns stdout, skips the human summary)
4. **`tests/mcp.test.ts`** — unit tests (mock stdin/stdout via dependency injection)
5. **README.md**, **docs/PRODUCT_PLAN.md** — docs

## Tool Specifications

### `get_repo_map`
- **Description:** "Returns the repository map: root path, package manager, detected frameworks, and shallow structure (directories, CI workflows)."
- **Input schema:** none (`{type: "object", properties: {}}`)
- **Returns:**
  ```json
  {
    "root": "/abs/path",
    "packageManager": "pnpm",
    "frameworks": ["next"],
    "structure": {
      "directories": ["src", "tests"],
      "ciWorkflows": [".github/workflows/ci.yml"]
    }
  }
  ```

### `get_commands`
- **Description:** "Returns detected package scripts and the subset used for verification (test/build/lint/verify)."
- **Input schema:** none
- **Returns:**
  ```json
  {
    "commands": [{"name": "build", "command": "tsc"}, ...],
    "verification": [{"name": "build", "command": "tsc"}, ...]
  }
  ```
  `verification` reuses `VERIFICATION_SCRIPTS = ["test","build","lint","verify"]` (exported from `render.ts` so both render and MCP share one source of truth).

### `get_readiness_notes`
- **Description:** "Returns agent readiness notes — early signals that may confuse humans or coding agents."
- **Input schema:** none
- **Returns:**
  ```json
  {"notes": ["README references script 'deploy' not in package.json", ...]}
  ```

### `refresh`
- **Description:** "Rescans the repository and refreshes the cached brief. Call this after files change so subsequent queries return fresh data."
- **Input schema:** none
- **Returns:**
  ```json
  {"scannedAt": "2026-06-17T12:00:00.000Z", "root": "/abs/path"}
  ```
  Updates the in-memory cache as a side effect.

## Protocol Layer Details

### JSON-RPC framing
- Newline-delimited JSON: one Request object per line on stdin; server writes one Response object per line on stdout
- Server reads via `readline.createInterface({input: process.stdin})`
- Server writes via `JSON.stringify(response) + "\n"` to `process.stdout`

### Methods handled

| Method | Behavior |
|--------|----------|
| `initialize` | Reply `{protocolVersion: "2025-06-18", capabilities: {tools: {}}, serverInfo: {name: "repo-brief", version: "0.4.0"}}` |
| `tools/list` | Reply `{tools: TOOL_DEFINITIONS}` |
| `tools/call` | Read `params.name`; dispatch via `executeTool`; on unknown tool return JSON-RPC error code -32601 |
| (other) | Return JSON-RPC error code -32601 (method not found) |

### Error handling
- Unknown tool name → JSON-RPC error `-32601` (method not found)
- Scan failure during `refresh` → JSON-RPC error `-32603` (internal error) with message
- Malformed JSON line → JSON-RPC error `-32700` (parse error)
- Startup scan failure → log to stderr and exit code 1 (no JSON-RPC possible, server can't initialize)

### Tool result format (per MCP spec)
`tools/call` returns `{content: [{type: "text", text: "<json-stringified-payload>"}]}`. The payload is JSON-stringified into the `text` field per MCP convention (tools return text content; agents parse structured data from it).

## Error Handling & Edge Cases

| Situation | Behavior |
|-----------|----------|
| Startup scan throws | Log to stderr, exit 1 (before JSON-RPC loop) |
| `refresh` scan throws | Return JSON-RPC error `-32603`; cache unchanged |
| Unknown tool | Return JSON-RPC error `-32601` |
| Malformed JSON line | Return JSON-RPC error `-32700` (id null since unparseable) |
| Empty stdin closed | Server exits cleanly (code 0) |
| Tool called before initialize | Not enforced (MCP clients always initialize first); tool reads cache regardless |

## Testing Strategy

### Unit tests (`tests/mcp.test.ts`, Node test style)

Tests inject `stdin` (a `Readable` / string stream) and capture `stdout` writes via a buffer, so no real process spawning.

1. **`initialize` → returns capabilities + 4 tools declared**
2. **`tools/list` → returns 4 tool definitions with correct names**
3. **`get_repo_map` → returns cached root/packageManager/frameworks/structure**
4. **`get_commands` → returns commands + filtered verification subset**
5. **`get_readiness_notes` → returns cached notes**
6. **`refresh` → rescans and updates cache (verify scannedAt changed / data refreshed)**
7. **Unknown tool → JSON-RPC error `-32601`**
8. **Malformed JSON line → JSON-RPC error `-32700`**

To make the server testable, `runMcpServer` accepts an optional `{stdin, stdout}` parameter (defaults to `process.stdin` / `process.stdout`). Tests pass a `PassThrough` stream as stdin (pre-filled with request lines) and capture stdout into an array.

### Manual acceptance (pre-release)

1. [ ] `echo '{"jsonrpc":"2.0","id":1,"method":"initialize"}' | repo-brief mcp` → returns capabilities
2. [ ] `echo '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' | repo-brief mcp` → returns 4 tools
3. [ ] `get_repo_map` returns the repo's own structure
4. [ ] Configure in Claude Code / Cursor, confirm agent can call tools

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| MCP spec drift (protocolVersion) | Low | Client compat issues | Pin to a dated version; can update in patch release |
| Agent can't parse our payload shape | Medium | Tool calls return confusing data | Return fields 1:1 with `RepositoryBrief` (familiar shape) |
| Blocking scan on startup delays first response | Low | Slow first tool call | Acceptable for stdio; agent tolerates init latency |
| `refresh` races with concurrent tool call | Low | Stale read | stdio is single-threaded request/response; no concurrency |
